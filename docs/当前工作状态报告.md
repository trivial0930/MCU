## 5. 当前限制与风险

当前还不能宣称已经完成最终上板闭环，原因如下：

- 本机缺少 `xc7k325tffg900-2` 器件支持或 license，无法完成 K7EDAEVAL 目标器件的最终实现矩阵。
- 当前综合冒烟未加载板级 XDC，不能代表最终引脚、时钟和 post-route WNS。
- 板上真实时钟目前仍来自 `CLK_50M`。95 MHz、100 MHz、110 MHz、120 MHz、130 MHz 目前只是 Vivado 时序目标，不代表板上已经真实运行这些频率。
- 若验收要求板上真实运行 95 MHz 以上，需要后续加入 PLL/MMCM 或外部高频时钟，并同步修改 `rtl/board_top.v` 与 `constraints/top.xdc`。
- 本地 Verilog 回归依赖 `iverilog` 和 `vvp`，脚本已经加预检查，但还需要在队友机器上安装后再跑。

## 7. 上板执行摘要

### 7.1 上板前硬件检查

真正连接板卡前，先完成以下检查：

| 检查项 | 要求 |
| --- | --- |
| 板卡型号 | 确认为 K7EDAEVAL，且 FPGA 型号与 `xc7k325tffg900-2` 一致。 |
| 供电 | 电源适配器、电源开关、板载电源指示灯正常。 |
| JTAG | 下载器连接稳定，Vivado Hardware Manager 能识别 FPGA。 |
| 时钟 | 板载 `CLK_50M` 可用；当前工程默认不会自动产生 95 MHz 以上真实时钟。 |
| 复位 | `KEY1` 作为复位输入时，按下/松开极性要与 `board_top.v` 中逻辑一致。 |
| 外设占用 | LED、按键、JTAG、时钟引脚不要被其他示例工程或约束复用冲突。 |

如果硬件管理器连不上，先不要改 RTL，优先检查 USB/JTAG 驱动、线缆、电源、Vivado Hardware Server 和板卡跳线/拨码设置。跳线/拨码的具体位置以实验板说明书为准，不在本文中硬写。

### 7.2 生成上板工程

首次上板建议进入：

```powershell
cd M:\routes\speed_v7_q7_narrow_mul\mcu_fft_q7_narrow_mul
D:\vivado\2025.2\Vivado\bin\vivado.bat
```

在 Vivado Tcl Console 中：

```tcl
set PART_NAME xc7k325tffg900-2
set TARGET_PERIOD_NS 20.000
set ENABLE_ILA 1
source ../../vivado/create_board_project.tcl
```

然后执行综合、实现和 bitstream：

```tcl
launch_runs synth_1 -jobs 4
wait_on_run synth_1
launch_runs impl_1 -to_step write_bitstream -jobs 4
wait_on_run impl_1
open_run impl_1
report_timing_summary
report_utilization
```

生成完成后至少保留以下结果给队友复查：

| 文件/报告 | 用途 |
| --- | --- |
| `board_top.bit` | 下载到 FPGA 的 bitstream。 |
| `board_top.ltx` | ILA probe 映射文件，仅启用 ILA 时需要。 |
| timing summary | 证明本次目标约束下 `WNS >= 0`。 |
| utilization report | 记录 LUT、FF、DSP、BRAM，便于和矩阵结果对照。 |

### 7.3 下载与首轮 ILA 验证

下载 bitstream 时，如果启用 ILA，需要同时选择 `.bit` 和 `.ltx` 文件。首次 ILA 建议这样触发：

| 项目 | 建议 |
| --- | --- |
| 触发信号 | 先用 `done == 1`；若看不到写出过程，再改用 `verify_we` 上升沿。 |
| 采样窗口 | 覆盖程序开始到 `done` 拉高后的若干周期。 |
| 复位方式 | 下载后手动按/松 `KEY1`，观察 `cnt_test` 是否重新计数。 |
| 判断顺序 | 先看 `done`，再看 `verify_we` 次数，最后核对 `verify_addr` 和 `verify_vector_out`。 |

首轮板上观察重点：

- `LED1` / `done` 最终拉高。
- `verify_we` 出现 16 次有效写入。
- 最后一笔 `verify_addr` 为 15。
- `cnt_test` 接近 157。
- `verify_vector_out` 写入顺序与 `mem/FFT_output.coe` 一致。

如果首轮不通过，建议按“硬件连接 -> 复位极性 -> 初始化文件 -> ILA 波形 -> 仿真回归”的顺序回退。不要先改算法 RTL，否则容易把板级问题和功能问题混在一起。
