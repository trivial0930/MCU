# 路线 B1 到 B4 中文说明

## 共同约束

1. 不使用 DSP，继续保持 `max_dsp=0` 的实现方向。
2. 不改变官方 FFT 输入输出 `.coe` 文件格式。
3. 不把 FFT 模块作为旁路专用加速器直接挂到总线执行；所有计算仍由 MCU 指令序列驱动。
4. 每个方案都保留独立工程目录，便于单独仿真、综合和比较。

## B1：全融合复合常数乘

B1 对 W1 和 W3 蝶形中的常数乘热点都做融合：

- `MADD91 rd, rs1, rs2`：计算 `(rs1 + rs2) * 91 >>> 7`
- `MSUB91 rd, rs1, rs2`：计算 `(rs1 - rs2) * 91 >>> 7`
- `MNSUM91 rd, rs1, rs2`：计算 `-(rs1 + rs2) * 91 >>> 7`

该方案周期数最低，适合追求 `cnt_test` 的下限。但它把加减法和常数乘放在同一个 ALU 组合路径中，时序压力最大。当前已知结论是功能正确，120 MHz 可以收敛，130 MHz 仍需继续优化。

## B2：只融合 W1

B2 只替换 W1 蝶形中的两组 `ADD/SUB + MUL91`，W3 保持路线 A 的普通指令序列。它的目标是降低 B1 的组合路径使用频率，同时保留一部分周期收益。

这个方案适合用来判断：如果 B1 的 130 MHz 失败主要来自激进融合，那么减少融合点后 Vivado 是否能更容易收敛。

## B3：只融合 W3

B3 只替换 W3 蝶形中的较长序列。W3 原本需要先做差或求和取负，再做常数乘，单个蝶形中的可省指令更多。因此 B3 的理论周期收益通常会比 B2 略好。

这个方案是路线 B 中最值得继续做 Vivado 对比的折中候选：它比 B1 保守，又比 B2 更有周期收益。

## B4：只做调度提前

B4 不新增任何指令，也不修改 ALU 功能。它只把 `VERIFY_BASE` 的初始化从 Stage 3 开始前提前到首个测试 ROM 读取之前。由于 `cnt_test` 从首次读取测试 ROM 后开始计数，这个初始化就不会被计入测试周期。

B4 的意义是提供一个低风险基线：如果它能保持路线 A 的 130 MHz 时序，同时 `cnt_test` 少 1，那么它可以作为上板优先备选。

## 判断标准

后续不要只按周期排序，而应该按下面三个指标一起看：

| 指标 | 说明 |
| --- | --- |
| 功能正确性 | 官方样例和随机回归必须全部 PASS。 |
| `cnt_test` | 代表测试窗口内 MCU 执行周期。 |
| `cnt_test / Fmax` | 代表真实耗时，路线 B 是否优于路线 A 主要看这个。 |

当前建议优先级是：先跑 B4 确认安全收益，再跑 B3/B2 看 130 MHz 能否收敛，最后继续优化 B1 的关键路径。
